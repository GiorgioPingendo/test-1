Github collaboration

Pingendo's collaboration is built on top of Github. 
Github knowledges are required to set up a collaboration project.
For contributors, Github is behind an easy interface that does not require any knowledge of Github.

Editor

The editor features dev and non-dev features. Users can choose to remain on the surface or to go deep according to everyone's role and skill. The WYSIWYG editor permits to all contributors to perceive real-time the effect of their modification. Some use cases:

  • Copywriters edit texts.
  • Photographers update links to photos.
  • Designers edit lay-out or styles.
  • SEO specialists add page title or description, revise texts and more... 
  • Marketers act where he/she can and distribute tasks.
  • Stakeholders can be main contributors or just be set "read only" and approve the job at required stages.
  • Developers focus on issues the above people can't. e.g. effects, custom responsive rules, deep styling...

Collaboration

1) What is a possible use case?

  • Developer set up the project in Github and invites the project manager as Github collaborator  
  • The project manager (PM) has a rough idea of what HTML is, so drags some sections to create a rough prototype about what's needed. 
  • PM adds some contents and the related photos.
  • PM asks to dev to fix some responsive issues.
  • PM asks to designer to add some CSS to improve esthetic.
  • Designer asks to developer some help on CSS.
  • PM asks to dev to add some functions in JavaScript
  • PM asks to dev to publish the page on Github pages
  • PM tests the online page
  • PM asks to dev to automatically push the master online.

2) What is a "Pull request"

It's just a copy of the original document (master) where you can add your contribution. it can:

  • Be closed (discarded) by the user who created it.
  • Be merged (accepted) by the project owner
  • Be closed (discarded) by the project owner
  • Every single file in the pull request can be compared to the master by any user.

3) How to create pull requests efficiently?

To help the project owner to revise your job and easily accept your contribution please remind to:

  • Give them a title related to the modification
  • Create a pull request for each different set of modification
  • Keep them small, with few modification each. It helps the PM to quickly understand the proposed modification.
  • Each pull request will be part of the powerful history of Github. Keeping them small means a granular "time machine" at your fingertips.
  • Comment on each pull request

4) How to manage contributors efficiently?

To help contributors to create in the required direction please remind to:

  • Revise contributions often
  • Comment

5) How to use comments efficiently?

Comments are great for details about a file or a pull requests. Please rely on your abitual project management tool to manage the whole project. (Trello, Asana, Basecamp...) 

6) What are the different roles?

  • Owner/Developer --> Usually the developer who has full control of the project and the users involved
  • Collaborator/Project manager--> Usually the marketer. He/she will manage the project by accepting/discarding pull requests and by distributing tasks.
  • User --> They can only make contribution by creating pull requests
  • Read only user --> They get only a link to the webpage without the possibility of contribution.

7) Do users have to signup to Github to submit pull requests?

Yes, because Github manages users and their actions. Don't worry, signing up to Github is just as easy as any other service and no more actions are required on Github.

    8) Does Pingendo save developer's time?

    Yes, we would like to. The Github integration permits to developers to manage a "safe environement" where users can iterate on content or source, each one at their own skill level.
    Our goal is to keep developers at fingertips of the marketing and communication team for technical issues only. Easy to reach, with clear requests related to real pages. 

    9) Does Pingendo save time to marketing and communication teams?

    We hope to. Pingendo is really effective on prototyping or when iteration is required on the whole page elements. When pages are too fluid for a traditional CMS.
    Screenshots, mockups, excel files with data, presentation, Word documents with texts and other means useful to explain things to developers or stakeholders, are not required any more.
    Iteration is taken straight to real pages, where it should be to make things simpler.

    9) How to invite contributors to public repositories?

    Send them the project link. The contributor has to login with Github to submit pull request.

    9) How to invite contributors to private repositories?

    • Ask contributors to signup to Github
    • Set them as collaborators
    • Send them the link to the project 

    9) What changes for Pingendo with private or public repositories or with personal or organization plan?

    Non of them makes any difference for Pingendo. Privacy settings and user settings are managed by Github only.

    9) I'm a developer and I have many private reposotories. How to grant access to Pingendo only to the project related repositories?

    Github API does not permit a granular access to repositories. 
    A developer can adopt an alternative method that really helps to keep sharing clean and simple.

    • Create a Github user e.g. "Webproject company abc" 
    • Grant access to that user to your repositories you would like to share
    • Grant access to Pingendo at the user "Webproject company abc".

    9) Is granting access to my repositories to Pingendo secure?

    Pingendo does not store any Github password. When you grant access to your repositories, your browser permits to Pingendo to talk to Github. No data are sent to Pingendo. (Pingendo has no servers out of the billing one)

    9) Who can access repositories?

    Pingendo does not manage permissions. Roles and permission depend of your Github settings.

    Who can read your source

    • Public repos --> everybody
    • Personal plans, private repos --> Collaborators
    • Organization plans, private repos --> Read permission and up

    Who can read your rendered pages?

    • Everybody who can read your source and has the link 

    Who can submit pull-requests by Pingendo?

    • Everybody who can read your source.
